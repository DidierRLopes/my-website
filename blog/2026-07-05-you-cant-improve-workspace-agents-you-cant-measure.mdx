A few weeks ago at [NY Tech Week](https://youtu.be/7fDTDYh2NJ4?t=1210) I showed agents doing real financial work inside OpenBB Workspace — reading data, building dashboards, assembling whole apps — driven over the [Workspace MCP](https://openbb.co/blog/introducing-workspace-mcp/) server. The demo ran on the [Stark Industries demo](https://github.com/DidierRLopes/stark-industries-demo), a complete but fake enterprise workspace.

A demo is a good way to show something *can* happen once. It is a bad way to know how often it happens, which agents can do it, or where they fall apart. If agents are going to drive this workspace for real analysts, that has to be measured, not vibe-checked from a stage. And by useful I mean the boring, concrete kind: can an agent inspect data, call the right tools, build a dashboard, repair broken state, and leave something behind that a human can open the next morning?

Nothing measured that, so I built [WorkspaceBench](https://github.com/DidierRLopes/openbb-workspace-bench). It includes the harness, a deterministic workspace simulator, 300 scenarios, the graders, and every number in this post. The Stark demo is also where the benchmark's enterprise scenarios come from — the same widgets and apps, turned into gradeable tasks.

This post covers the benchmark end to end: how a workspace task becomes a scored episode, how the scenarios are generated and certified, and what real models score.

## Why a benchmark {#why}

The idea in this section is simple: grade agent work the way we grade software, and judge a workspace agent by the state it leaves behind.

### The pattern, elsewhere

An eval bench is a repeatable test suite for agents. Instead of saying "this model feels better", you give every agent the same task, the same tools, the same starting state, and the same grader.

The inspiration is the pattern. [SWE-bench](https://www.swebench.com/) ([Jimenez et al., 2023](https://arxiv.org/abs/2310.06770)) made software issues measurable. [Terminal-Bench](https://www.tbench.ai/) made terminal work measurable. [WebArena](https://webarena.dev/) ([Zhou et al., 2023](https://arxiv.org/abs/2307.13854)) and [OSWorld](https://os-world.github.io/) ([Xie et al., 2024](https://arxiv.org/abs/2404.07972)) pushed the same idea toward browser and desktop tasks. WorkspaceBench is my attempt to bring that discipline to financial workspaces.

### A chat answer is not enough

Most agent demos are still judged by whether the final answer sounds good. That is fine for conversation and weak for real workflows. In a workspace, the important question is not what the agent says. It is what the agent changed.

Did it create the correct widget? Did it use the right ticker? Did it inspect the schema before creating anything? Did it repair the dashboard without breaking the layout? A software agent is judged by whether the tests pass, not by how confident the diff description sounds. A workspace agent should be judged the same way: by the dashboard, tabs, widgets, layout, and notes it leaves behind.

So WorkspaceBench grades final state deterministically — exact widget identifiers, exact data arguments, layout bounds, overlap checks — before it considers anything the agent said, the same final-state-first stance [τ-bench](https://arxiv.org/abs/2406.12045) takes for tool agents.

One thing worth being precise about:

> WorkspaceBench measures whether an agent can operate the workspace, not whether it can reason about markets.

<br />

The fixture data is deterministic and fake on purpose. The finance framing lives in the task language — mapping "prep the earnings review" onto the right widgets is a workspace skill — but no scenario is graded on investment insight. That keeps the reward signal clean, and it is also what lets the same grader later serve as a training reward.

That clean reward has an obvious edge case: content checks are deterministic fact-presence checks, so a note passes when it mentions the required ticker, EPS, or revenue. A keyword-stuffing agent could game that layer. I accept that tradeoff deliberately: no LLM judge variance, a clean training reward, and trace checks such as schema-before-create, invalid-call budgets, and `missing_tool_result` requiring the agent to read the data it cites close the cheapest stuffing paths. Reward robustness gets stress-tested the day the grader becomes an RL reward; the trace checks are the first line of defense for that, too.

That is one scenario. The rest of this post is how I turn it into 300 — same contract, same grader — and see which agents hold up. Here is the whole suite at a glance:

- **300 scenarios** — one release, all generated the same way, all certified the same way, all equal weight.
- **15 families × 5 tiers × 4** — every family anchored on one MCP capability with a complete t0–t4 ladder.
- **20 MCP tools + prompts + resources** — from `get_widget_schema` to `read_workspace_resource`; every one exercised by ≥20 scenarios.
- **180 / 60 / 60 splits** — train/validation/test, stratified per family×tier cell, not by provenance.
- **Release gates: oracle 300/300 · no-op 0/300** — plus quota checks (backend balance, L2 share, tier balance, 300 unique structural fingerprints) enforced in CI.

## Inside the benchmark {#inside}

How a task becomes a score: the loop, the surface the agent lives on, the kinds of work, and the contracts that hold it together.

### From workflow to scored episode

The core loop is deliberately small. A scenario defines the task. The harness seeds a deterministic workspace from fixture backends. The agent emits tool calls. The simulator applies them. The grader checks the final state and the trace.

The simulator is intentional. It is not a pixel or browser simulator; it is a deterministic state machine for the Workspace MCP contract ([`src/workspace_bench/workspace/simulated_workspace.py`](https://github.com/DidierRLopes/openbb-workspace-bench/blob/main/src/workspace_bench/workspace/simulated_workspace.py)). That makes runs fast, reproducible, and cheap enough for CI. A live smoke runner ([`src/workspace_bench/workspace/live_mcp.py`](https://github.com/DidierRLopes/openbb-workspace-bench/blob/main/src/workspace_bench/workspace/live_mcp.py)) drives the same scenario contract through a real local `workspace-mcp` sidecar and websocket bridge, with the benchmark simulator acting as the browser side. A separate read-only audit lists the hosted deployment's tools, prompts, and resources and diffs that surface against what the scenarios assume.

## A designed distribution {#distribution}

The 300 scenarios are generated as a lattice, not collected one by one. And the difficulty axis had to earn its shape empirically before I trusted any of the results below.

### Generated, not hand-written

Every scenario in the benchmark is generated — including the early ones I drafted myself, which were AI-written too, so pretending they were a special hand-audited caste would be dishonest. What matters is not who typed the JSON; it is whether the distribution is designed. My first pack quietly skewed: four MCP tools appeared in exactly one scenario each, and half the suite tested the same single-widget motion. The lesson from [TMax](https://wai-org.com/blog/tmax/) — 14,600 terminal environments sampled compositionally from nine structured axes — is that a task distribution should be a designed property, not an accident of what the author felt like writing.

So the suite is generated compositionally: **15 families, each anchored on one MCP capability, each carrying a complete t0–t4 ladder** — 4 scenarios per rung per family, 300 total. The same tool is tested at every difficulty. `delete_widget` at t0 means "remove the only widget"; at t4 it means "remove one duplicate, fix the survivor, document the repair" in a single episode. Tiers are not the L0–L4 levels from earlier — levels classify the kind of work, tiers grade how structurally demanding the episode is, and a single-widget L1 task can sit anywhere from t0 to t4.

Tiers are structural, not adjectives. A tier is defined by what the task is made of — composition, seeded pathologies, discovery pressure, turn budget — and it gets harder from t0 to t4 by construction. The composition is measurable: the average episode uses 2.3 tools at t0 and 5.0 at t4.

Splits are built in: 180 train / 60 validation / 60 test, stratified per family×tier cell so every slice of the distribution appears in every split. One honest caveat for anyone training on this: stratification means the same template family spans splits with different parameters — for evaluation that is exactly right, but if you train on the train split, prefer held-out-family splits to rule out template-shaped shortcuts. All 300 scenarios ship publicly, including the test split, so the split is a bookkeeping device, not contamination protection. The real defense is that the generator is deterministic and template-driven, so fresh held-out packs can be regenerated at will. Reproduce or extend it:

```bash
# regenerate the pack (deterministic templates, no LLM required)
python scripts/generate_gen_pack.py

# certify: oracle passes, no-op fails, quotas + fingerprints hold
uv run --extra dev workspace-bench validate \
  --pack all --min-scenarios 300

# rebuild the per-test tool matrix
python scripts/generate_tool_matrix.py
```

### Calibration: how a test earns its tier

A tier is only real if the harder tiers are actually harder. So before trusting any score, I calibrated the difficulty axis empirically. Pick one mid-strength model — gpt-4.1-mini, cheap enough to run a lot — run it over all 300 scenarios, and require its pass rate to fall monotonically from t0 to t4. If the curve comes out flat or bumpy, the tier labels are decoration and the generator needs fixing. If it falls cleanly, the tiers are a real ladder and every test in them has earned its label.

When you generate tasks instead of hand-writing them, you cannot trust any single one — the generator will produce broken tasks, unsolvable tasks, and tasks that are secretly trivial. This is the problem [TMax](https://wai-org.com/blog/tmax/) runs into at 14,600 environments and solves with a pipeline: generate, then put every candidate through machine checks before it is allowed into the suite. WorkspaceBench borrows that idea. Each scenario has to clear a gauntlet before it counts, and the whole thing runs as a loop:

  {() => {

    return <CalibrationLoopDiagram />;
  }}
</BrowserOnly>

Read it left to right. **Generate** a candidate from a family template. **Certify** it mechanically: the reference solution has to pass the grader and a no-op agent has to fail it — that alone rejects unsolvable and trivially-passable tasks. **Gate** it by running the calibration model and checking it lands in the pass-rate band its tier claims. If a whole tier or family comes out wrong, that is a bug in the **generator**, not in one task — so you fix it there and re-run the loop over all 300. Nothing enters the benchmark by hand; it enters by surviving.

That loop earned its keep immediately. The first gating run exposed three real defects in my own benchmark: two brand-new families scored 0/20 because the harness never documented the two new tools' schemas to models (so they invented argument shapes), and because several prompts demanded facts the grader checked but the instructions never asked for. A third pass caught an inconsistency in when schema-discipline was instructed versus silently expected. All three were fixed at the generator level, the pack re-certified, and the sweep re-run from scratch — and the gating curve went from a broken 83–67–48–47–30 to the clean **98–92–77–52–33** above. That is the difference between a benchmark and a demo: a demo shows the good runs, a benchmark survives its own generator being wrong.

## Results {#results}

The first useful question is not whether one model produced a nice answer once. It is whether different agents can solve the same workflows under the same constraints, and whether their failures are understandable.

These are real runs from the interactive comparison harness ([`src/workspace_bench/reports/model_compare.py`](https://github.com/DidierRLopes/openbb-workspace-bench/blob/main/src/workspace_bench/reports/model_compare.py)): each model chooses one tool call, receives the simulated workspace result, then chooses the next call — all six against the same 300 certified scenarios, same grader, same logs. Every episode's full transcript is exported as portable rollout JSONL in [`runs/exports/`](https://github.com/DidierRLopes/openbb-workspace-bench/blob/main/runs/exports/).

Six models, chosen to span the range from frontier down to what runs on a laptop:

- **GPT-5.5** — the most advanced model available.
- **Claude Sonnet 5** — a recent release, and fairly affordable.
- **GLM-5.2** — the best open-weights model right now.
- **gpt-4.1-mini** — a solid, cheap model, and the gating model that calibrated the tiers.
- **gpt-oss:20b** — a strong open-weights mixture-of-experts.
- **Qwen3 8B** — a small dense model.

The last two run on my own machine through Ollama for free; the rest go through vendor APIs and OpenRouter.

The rows nobody passes are the most valuable output of the whole sweep. 15 of 300 scenarios defeat all six models, and they are not scattered: 8 are inspect-family tasks (deduplicate or reconcile a workspace the agent did not build — ambient-state reasoning), 4 are companion-widget parameter tasks, 2 are precise layout splits, and 1 is a full skills-driven tearsheet. Even GPT-5.5 passes only 11/20 of the inspect family. That list is where a future t5 comes from, and it doubles as a work queue for the tool surface itself. It is also the clearest evidence that the remaining headroom is about reading state, not composing calls. At the opposite end, 93 scenarios are passed by everyone — the sanity floor, 70 of them in t0–t1, which is what "t0 nearly free" should look like.

Failures are also model-specific in ways a single number hides. Qwen3 8B scores 4/20 on the apps family — long instantiation chains — where every other model scores 17 or better. GLM-5.2 is the only model that struggles with the prompts family (8/20), because it skips the prompt fetch it considers unnecessary. gpt-oss:20b bottoms out on documentation notes (5/20). Six models, six different failure fingerprints on identical tasks — which is why the chart below breaks every model out by failure type, instead of averaging the differences away.

> The honest read: the benchmark separates the mid-field sharply, but the frontier is already near the ceiling, and what it still fails is precision and ambient state, not composition. I would treat that as the roadmap for t5.

## What the score is actually for {#cuts}

A leaderboard is the least interesting thing here. The reason to grade agents this precisely is that every failure points at something you can fix — and it points in two directions.

One is the **tool surface**. Look at the failure analysis again: GLM-5.2 tripped the invalid-call budget on dozens of scenarios, and `schema_not_called_before_create` dominates the mid-field. Those are not only verdicts on the models; they are verdicts on the MCP tools — the schemas, the names, the error messages the workspace hands to agents. The same graded run that ranks the models is a to-do list for the server: make the surface clearer and every agent gets better at once. That is a benchmark I can point back at the product.

The other is the **agents themselves**. Every episode leaves a full, graded trace — 1,800 of them are already in the repo. A grader that emits a clean pass/fail signal is exactly what a reinforcement-learning loop needs for reward, and the same deterministic simulator can serve as the environment. Turning these traces into training signal for smaller or cheaper agents is the obvious next step.

Both of those are open work, and both reuse this same benchmark unchanged. For now the point stands on its own: if agents are going to drive a financial workspace, you have to be able to measure how well they do it — and then the measurement tells you what to improve. Incentives always win. The least I can do is measure the ones I am creating.
